- **Cloud Functions (`functions/package.json`)**: 1 real finding — a moderate-severity issue in the

## 2. Dynamic Application Security Testing (DAST)

**Tool:** OWASP ZAP (Zed Attack Proxy) v2.16, official Docker image
(`ghcr.io/zaproxy/zaproxy:stable`), baseline passive scan.

**Target:** the live, publicly deployed production application at `https://parabola-6a220.web.app`
(first deployment of the frontend to Firebase Hosting was performed as part of this review, so this is
a genuine external scan of the real production surface, not localhost).

**Command:** `zap-baseline.py -t https://parabola-6a220.web.app`

### Initial scan (before remediation)

- **0 FAIL, 11 WARN, 56 PASS.**
- Findings: missing Content-Security-Policy, X-Content-Type-Options, X-Frame-Options,
  Permissions-Policy headers; missing Subresource Integrity attributes; missing
  Cross-Origin-Embedder-Policy; cacheable static content; a Unix-timestamp-shaped string in the JS
  bundle.
- Full report: `security/scans/zap-report/` (regenerated by the second run below; the initial run's
  findings are summarized here and in the remediation log in `VULNERABILITY_MANAGEMENT.md`).

### Remediation applied between scans

Added security headers via Firebase Hosting configuration (`firebase.json`): Content-Security-Policy
(scoped to the actual domains the app depends on — Google Fonts, Firebase/Google Cloud APIs, Zoom's
account sign-in popup, Firebase Storage), X-Content-Type-Options, X-Frame-Options, Referrer-Policy,
Permissions-Policy, Cross-Origin-Opener-Policy (`same-origin-allow-popups`, chosen specifically to
preserve the app's Google Sign-In popup flow), and Strict-Transport-Security.

The new CSP was verified against the live deployment with real end-to-end flows (account signup,
Google/email authentication paths, Firestore reads/writes, roster and class creation) to confirm no
functional breakage before being considered final — see `security/scans/csp-verification-notes.md`.

### Re-scan (after remediation)

- **0 FAIL, 8 WARN, 59 PASS.**
- Full report: `security/scans/zap-report/zap-baseline-report.html` /
  `zap-baseline-report.json`.
- Remaining WARN items and why each is accepted, deferred, or a reviewed false positive are documented
  in `security/policies/VULNERABILITY_MANAGEMENT.md`, Section 6 ("Accepted risks / deliberate
  non-fixes"). None represent a FAIL-level or actively exploitable finding.
